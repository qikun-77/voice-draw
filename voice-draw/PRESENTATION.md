## 第二部分：问题定义（0:30 - 1:00）

🎤 **"做这个项目的原因很简单——你有没有见过这样的场景：设计师灵感来了想画草图，但手边没触控笔；视障用户想表达视觉创意却没法操作画板；或者你只是想边做饭边口述画个便签。现有的绘图工具都需要鼠标或触屏，还没有一款能用自然语言直接创作的产品。"**

🎤 **"所以我想解决的核心问题是——让绘图的门槛降到最低，让人与画布的交互变成对话。"**

---

## 第三部分：技术架构（1:00 - 2:00）

🖱️ 切换到 VS Code，展示 `server.py` 和 `index.html` 的文件结构。

🎤 **"架构分三层。"**

🎤 **"第一层是交互层——浏览器端。我用了 Web Speech API 做中文语音识别，Web Audio API 采集麦克风音频驱动波形动画。渲染层用的是 SVG DOM，选择 SVG 而不是 Canvas 的原因很简单——每个图形是独立的 DOM 元素，天然支持 id 标识、属性修改和删除，这让后面的指代消解和状态管理容易很多。"**

🎤 **"第二层是解析层——Python FastAPI 后端。核心设计是混合模式：本地规则引擎加 LLM 降级。什么意思呢？当用户说'画一个红色的圆'，本地引擎用正则加关键词匹配直接返回 JSON，零 API 消耗。当用户说'画一个奥特曼'这种复杂指令，本地引擎无法处理，才调用 DeepSeek API 进行语义理解。"**

🎤 **"第三层是 AI 层——DeepSeek 大模型。我写了一个非常详细的 System Prompt，里面定义了操作类型、颜色映射、空间布局参考、尤其是九种复杂角色的几何分解方案。"**

🖱️ 滚动展示 System Prompt 中角色分解的部分。

🎤 **"比如奥特曼，我在 Prompt 里写死了它应该由银色椭圆头加红色椭圆身体加黄色圆眼加四肢组成。LLM 拿到这个模板，生成的就是具体的坐标和颜色，而不是抽象的描述。"**

---

## 第四部分：核心亮点（2:00 - 3:00）

🎤 **"这四点是我想强调的亮点。"**

**亮点一：混合引擎的「零消耗」设计**

🎤 **"我做了一组数据统计——常见的绘图指令中，约 85% 可以被本地规则引擎直接处理。包括 11 种基本图形和 4 种复合图形。只有复杂角色分解和指代消解才走 API。最频繁的'画圆'、'清空'、'撤销'这些操作，完全不消耗一分钱。"**

**亮点二：指代消解**

🎤 **"这是自然交互必须解决的问题。你可以说'把它变大'、'把那个红色的删掉'、'移动第二个圆'——系统需要理解'它'是谁、'那个红色的'是谁。我的做法是在前端维护一个对象状态池，每次请求把画布上所有对象的类型、颜色、id 发给 LLM，由 Prompt 规则做匹配。本地引擎也做了一层简单消解——按形状类型和颜色匹配最近创建的对象。"**

**亮点三：复杂角色几何分解**

🎤 **"这是 LLM 发挥创造力的地方。刚才说的奥特曼、钢铁侠、小黄人——我给 Prompt 的是约束，给 LLM 的是自由度。它需要在这些约束下生成精确坐标。这里很考验 Prompt 工程能力——你要告诉模型画布尺寸是 800 乘 600，圆半径不能超过 150，颜色必须用十六进制，否则渲染会崩。"**

**亮点四：完整的 Git PR 规范**

🎤 **"整个项目拆分成六个独立 PR，每个 PR 只做一件事——骨架、引擎、UI、形状、角色、交互——合并后主分支始终可运行。PR 记录文件里写了每个 PR 的标题、功能描述、实现思路和测试方式。"**

## 第六部分：难点与解决（4:30 - 5:15）

🎤 **"说几个开发中遇到的难点。"**

**难点一：System Prompt 如何让 LLM 输出稳定的结构化 JSON**

🎤 **"DeepSeek 有时候会在 JSON 外面包 Markdown 代码块标记，或者坐标超出画布。我在后端加了字符串清理逻辑，温度设到 0.1，Prompt 里反复强调'严格 JSON，无额外文本'。即便如此，实际部署前我跑了大概二十条测试用例确保输出符合预期。"**

**难点二：CORS 和 file:// 协议的限制**

🎤 **"最初设计是双击 index.html 打开前端，用 fetch 请求 localhost API。但浏览器安全策略禁止 file:// 页面向 localhost 发请求，CORS 配了也没用。最后解决方案是让 FastAPI 用 StaticFiles 托管前端文件，统一通过 http://localhost:8765 访问。这个改动花了我不少调试时间。"**

**难点三：语音识别的浏览器兼容性**

🎤 **"Web Speech API 在 Chrome 和 Edge 上可用，Firefox 不支持。中文识别在噪声环境下的准确率也有波动。目前的降级方案是——如果不支持语音，快捷按钮面板仍然可用。未来有计划接入更专业的 ASR 服务。"**

## 第七部分：总结与展望（5:15 - 5:45）

🎤 **"总结一下。"**

📊 **关键数据：**
- 支持 **11 种基本图形** + 4 种复合图形 + 9 种角色分解
- **85% 指令本地处理**，零 API 消耗
- 端到端延迟：语音识别 0.5-2 秒 + 解析 <100ms（本地）或 1-3 秒（API）
- **6 个独立 PR**，主分支始终可运行

🎤 **"未来我会继续完善三个方向：一是接入讯飞或 Azure 语音服务提升识别准确率，二是增加图层拖拽和自由变换，三是支持多人协作语音绘图。"**

🎤 **"这个项目让我深刻体会到——大模型不是银弹。混合架构才是落地的关键：高频简单任务走规则，低频复杂任务调 AI。这套思路可以用在很多场景。谢谢老师们！"**
